RAG sobre tu Knowledge Base de Obsidian

Arquitectura para hacer preguntas en lenguaje natural sobre tus 1,429 archivos de Obsidian (transcripciones de podcasts, notas de proyectos, conocimiento financiero) y obtener respuestas contextualizadas con referencias a las fuentes originales.

Como funciona paso a paso

1. Leer .md files -> 2. Partir en chunks -> 3. Convertir a vectores -> 4. Guardar en DB -> 5. Buscar por similitud -> 6. LLM responde con contexto

RAG = Retrieval Augmented Generation. En vez de meter todos tus 1,429 archivos en el prompt del LLM (imposible, serian ~16MB de texto), buscas primero los fragmentos mas relevantes y solo esos se pasan al modelo.

Tu caso concreto: 1,429 archivos x ~1,500 palabras media = ~2.1M palabras = ~2.8M tokens. Incluso con un context window de 1M tokens, no cabe todo. Con RAG, cada query solo necesita ~5-10K tokens de contexto relevante.

Opcion B: RAG completo con Python (recomendada)

Esta es la opcion mas potente y la que te da control total. Aqui esta el stack completo:

Paso 1 — Ingest

Leer y parsear Obsidian

Script Python que recorre tu vault, lee cada .md, extrae el frontmatter YAML (titulo, fecha, podcast, tags, url) y el contenido.

Python pathlib python-frontmatter watchdog (live sync)
Paso 2 — Chunk

Partir en fragmentos semanticos

Dividir cada archivo en chunks de ~500 tokens respetando limites naturales (headers, parrafos). Cada chunk hereda los metadatos del archivo padre.

langchain text_splitters tiktoken RecursiveCharacterSplitter
Paso 3 — Embed

Convertir texto a vectores

Cada chunk se convierte en un vector de 1536 dimensiones usando un modelo de embeddings. Textos semanticamente similares quedan cerca en el espacio vectorial.

OpenAI text-embedding-3-small o Voyage AI o sentence-transformers (gratis, local)
Paso 4 — Store

Guardar en base de datos vectorial

Los vectores + metadatos se guardan en ChromaDB (local, SQLite). Soporta filtros por metadata: podcast, fecha, tags.

ChromaDB o LanceDB o Qdrant SQLite backend
Paso 5 — Query

Buscar por similitud

La pregunta del usuario se embede con el mismo modelo. Se buscan los Top-K chunks mas cercanos por cosine similarity. Se pueden filtrar por metadata (solo Latent Space, solo 2026, etc).

cosine similarity Top-K = 10 metadata filters reranking (opcional)
Paso 6 — Generate

LLM responde con contexto

Los chunks recuperados se inyectan en el prompt del LLM junto con la pregunta. El modelo responde citando las fuentes (archivo, fecha, podcast).

Claude API (Sonnet/Opus) system prompt con instrucciones source citations

System prompt para el LLM

SYSTEM: Eres un asistente experto que responde preguntas basandose UNICAMENTE
en los fragmentos de la knowledge base proporcionados.

REGLAS:
1. Responde SOLO con informacion de los fragmentos. Si no hay info relevante, dilo.
2. Cita las fuentes: [Podcast, Fecha, Episodio]
3. Si multiples fuentes coinciden, sintetiza y compara.
4. Responde en el idioma de la pregunta.

CONTEXTO RECUPERADO:
{chunks recuperados por similarity search}

PREGUNTA: {pregunta del usuario}

Mejoras avanzadas

Mejora 1

Hybrid Search

Combinar busqueda semantica (embeddings) con busqueda lexica (BM25/keyword). Mejora recall para terminos tecnicos exactos como "STS" o "Basel III".

BM25Reciprocal Rank Fusion
Mejora 2

Reranking

Despues de recuperar Top-20 chunks, usar un modelo de reranking (Cohere, BGE) para reordenar por relevancia real. Mejora precision significativamente.

Cohere RerankBGE Reranker
Mejora 3

Multi-Query

Generar 3-5 variaciones de la pregunta del usuario con un LLM, buscar con cada una, y fusionar resultados. Captura matices que una sola query pierde.

Query expansionHyDE
Mejora 4

MCP Server

Exponer el RAG como un servidor MCP que Cursor, Claude Code, u OpenClaw pueden usar como herramienta. Asi cualquier agente puede consultar tu knowledge base.

MCP protocolstdio transport
